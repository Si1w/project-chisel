# Commands

Status: Accepted for v0 command direction

## Problem

Chisel's source code is the TOML project tree, not Rust code generated by
the agent. The command surface must therefore behave more like a compiler
toolchain than a collection of ad hoc runtime shortcuts.

Current commands mix four different responsibilities:

- editing source artifacts (`game.toml`, `entities/`, `scenes/`,
  `rules/`, `input.toml`);
- validating and compiling those artifacts into a runnable game plan;
- running an instantiated engine state;
- controlling an already-running session.

If these stay mixed, agents cannot reliably answer "is the game definition
valid?" before asking a human to review pixels. Human review would catch a
visual problem, but the agent needs compiler-style diagnostics and JSONL
runtime records to locate the source artifact that caused it.

## Toy Version

The first useful version is a command classifier plus a minimal compiler
entrypoint:

```text
chisel compile [root]
chisel compile [root] --format jsonl
```

`compile` validates the TOML project and proves it can be lowered into a
runnable game plan without running game time. The first implementation
reuses the existing bootstrap/load path as the lowering proof, then
reports compiler-style diagnostics.

The toy skips build artifacts, incremental compilation, source spans, and
visual rendering. It is useful alone because it gives the agent a fast
"is this game complete enough to run?" gate.

## Premises

1. TOML is the source language. CLI commands should be evaluated by
   whether they edit, validate, compile, instantiate, or control that
   source.
2. `run` and `play` should not be responsible for explaining source
   mistakes. They may call `compile` internally, but diagnostics belong
   to the compiler layer.
3. One-shot commands like `step`, `inspect`, and `emit` are convenience
   wrappers, not the core architecture. The core runtime control surface
   is JSONL session commands.
4. Human review and agent debugging are complementary. Humans inspect the
   rendered game; agents inspect compiler diagnostics and runtime JSONL.

## Command Classes

Every command must fit exactly one primary class.

| Class | Mutates TOML? | Instantiates runtime? | Persistent state? | Primary output |
| --- | --- | --- | --- | --- |
| Source authoring | Yes | No | No | TOML files / stderr diagnostics |
| Compiler | No | No | No | diagnostics |
| Runtime launcher | No | Yes | Yes | runtime JSONL / window |
| Session command | No | Already running | Yes | JSONL records |
| Convenience wrapper | No | Usually ephemeral | No | same as underlying command |

### Source Authoring Commands

These commands edit source artifacts. They must not tick the engine.

```text
chisel new <dir>
chisel add entity <name> ...
chisel add scene <name>
chisel add rule <id> ...
chisel add input <id> ...
```

Agent direct file edits are equally valid. Authoring commands exist to
make common TOML mutations safer, not to become the only authoring path.

### Compiler Commands

These commands validate source artifacts. They must not run game time.

```text
chisel compile [root]
chisel compile [root] --format human|jsonl
```

Future reserved commands:

```text
chisel build [root] --out <dir>
chisel explain <diagnostic-id>
```

`build` is reserved for when Chisel has a durable compiled artifact. v0
should not invent one prematurely. Until then, `compile` is the compiler
front door: it validates and lowers TOML source in memory.

### Runtime Launcher Commands

These commands compile/instantiate the TOML source and then run an engine.

```text
chisel run [root] [--dt 0.016] [--max-ticks N]
chisel play [root] [--dt 0.016]
```

`run` is headless. It writes JSONL records.

`play` is native human review. It opens a window, but it still writes
JSONL observation records so the agent can correlate human feedback with
runtime facts.

### Session Commands

These are JSONL records sent to an existing runtime session, not primary
CLI subcommands.

```jsonl
{"channel":"command","type":"step","count":1}
{"channel":"command","type":"inspect","query":null}
{"channel":"command","type":"simulate_input","event":{"type":"key_press","key":"Space"}}
```

Future session commands:

```jsonl
{"channel":"command","type":"pause"}
{"channel":"command","type":"resume"}
{"channel":"command","type":"reload"}
{"channel":"command","type":"save"}
```

Runtime commands should return `command-ack` records rather than process
exit codes because the process stays alive.

### Convenience Wrappers

These commands may remain for developer ergonomics, but docs should label
them as wrappers over a short-lived session, not as the core model.

```text
chisel step [root] [N]
chisel inspect [root] [--query QUERY]
chisel emit <input-event-json> [root]
```

They should be equivalent to:

```text
bootstrap source -> run one session command -> write output -> exit
```

They should not gain behavior that the JSONL session command cannot
express.

## Decision Checklist

When adding or redesigning a command, answer these in order:

1. Does it modify TOML source?
   - Yes: source authoring command.
2. Does it validate TOML without advancing time?
   - Yes: compiler command.
3. Does it instantiate `EngineState` and keep it alive?
   - Yes: runtime launcher.
4. Does it require an already-running engine state?
   - Yes: session command JSONL.
5. Is it only a shortcut for one of the above?
   - Yes: convenience wrapper. Keep it thin.

If a proposed command answers "yes" to more than one primary class, split
it.

## Diagnostic Model

Compiler diagnostics should be structured, source-oriented, and stable
enough for agents to act on.

```json
{
  "channel": "diagnostic",
  "level": "error",
  "code": "unknown-template",
  "path": "scenes/main.toml",
  "message": "scene references unknown template \"Player\""
}
```

Minimum fields:

- `level`: `error`, `warning`, or `note`;
- `code`: stable kebab-case identifier;
- `path`: project-relative source path when available;
- `message`: concise human-readable explanation.

Nice-to-have later:

- source span line/column;
- related diagnostics;
- machine-readable labels such as `template`, `rule`, `param`.

## Compiler Pipeline

The intended conceptual pipeline is:

```text
TOML project
  -> parse schemas
  -> validate references and names
  -> compile into GamePlan
  -> instantiate EngineState
  -> run/play/session
```

Suggested future Rust shape:

```rust
pub struct Diagnostic {
    pub level: DiagnosticLevel,
    pub code: String,
    pub path: Option<PathBuf>,
    pub message: String,
}

pub struct CompileReport {
    pub diagnostics: Vec<Diagnostic>,
}

pub struct GamePlan {
    // validated templates, scene, rules, input mapper config, physics config
}

pub fn compile_project(root: &Path) -> Result<GamePlan, CompileReport>;
pub fn instantiate(plan: GamePlan) -> Result<EngineState>;
```

v0 may keep using `bootstrap` internally, but new command design should
name the compile boundary explicitly.

## Proposed v0 Command Surface

Primary commands:

```text
chisel compile [root] [--format human|jsonl]
chisel run [root] [--dt 0.016] [--max-ticks N]
chisel play [root] [--dt 0.016] [--width 800] [--height 600]
```

Source authoring commands can remain planned, but `compile` should land
before expanding them:

```text
chisel new <dir>
chisel add entity ...
chisel add scene ...
chisel add rule ...
chisel add input ...
```

Convenience wrappers:

```text
chisel step [root] [N]
chisel inspect [root] [--query QUERY]
chisel emit <input-event-json> [root]
```

Documentation should present the primary flow as:

```text
edit TOML -> chisel compile -> chisel run/play -> inspect JSONL/human feedback -> edit TOML
```

## Validation Scope for `compile`

First implementation should validate:

- `game.toml` exists and schema version is supported;
- `main_scene` resolves to `scenes/<name>.toml`;
- entity templates parse and have unique names;
- scene entity template references resolve;
- component names are supported in v0;
- scene initial events are object-shaped with string `type`;
- rule files parse and each rule has at least one action;
- rule match params and action param references are valid;
- `spawn.template` action references a known template;
- `input.toml` parses when present;
- input payload substitutions reference existing input match params;
- tag interning stays under the v0 tag cap.

Warnings can be added after errors:

- unused template;
- rule event never emitted by built-ins, input, or initial events;
- input map that emits an event no rule consumes.

## Relationship to Render

`play` must not bypass the compiler or runtime boundaries. It should run:

```text
compile TOML
  -> instantiate EngineState
  -> winit/pixels frontend
  -> InputEvent -> InputMapper
  -> Schedule + RuleProcessor
  -> draw World
  -> write JSONL observation records
```

Human review happens through pixels. Agent debugging happens through
diagnostics and JSONL. Both describe the same compiled TOML source.

2. Implement `chisel compile [root] --format human|jsonl` with

4. Design `play` only after `compile` exists, so native rendering cannot
